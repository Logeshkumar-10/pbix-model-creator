## Step 0 — Determine Dataset Shape (Always Run First, Before Mode Detection)

This skill builds two structurally different kinds of model, and getting this wrong produces a BIM that deploys but models the wrong thing. **Decide this before anything else** — before asking mode questions, before reading any schema in detail.

A dataset is **Planning-shaped** if it has, or is meant to have:
- A scenario dimension (Actual / Plan / Forecast / Budget, or similar)
- An account hierarchy (chart-of-accounts style rollup feeding a P&L)
- A financial fact table at account × period × org-unit grain
- Variance, YTD/QTD, EBITDA-style, or scenario-comparison reporting as the primary use case

A dataset is **Non-Planning-shaped** if it instead looks like:
- A chart-feature showcase (bubble/scatter/quadrant demos, positioning maps)
- Behavioural, engagement, or correlation data with no scenario or account dimension
- A small number of flat or near-flat fact/dimension tables built to drive specific visuals rather than a full P&L

**How to decide:**
1. If a dataset-creator skill already ran in this conversation, check what it produced — `DimScenario.csv` / `DimAccount.csv` / `FactFinancial.csv` present → Planning. Two or three flat tables with no scenario/account structure → Non-Planning.
2. If a PBIX or description doc is uploaded, skim table names before asking anything — `[Dim] Scenario`, `[Dim] Account`, `[Fact] Financial` (or equivalents) → Planning. Otherwise → Non-Planning.
3. If neither is clear from context, ask directly:

```
Before I build this, one thing decides the whole approach:

Is this a planning/financial dataset (Actual vs Plan vs Forecast, account
hierarchy, EBITDA-style KPIs) — or a non-planning dataset (chart-feature
showcase, behavioural/correlation data, or something else without a
scenario/account structure)?

A) Planning / financial
B) Non-planning / chart-showcase
```

**Once decided, load the matching reference file before doing anything else:**
- Planning → read `references/planning.md` in full. It contains the Mode 1 intake steps, relationship map, measure architecture, and table/column description catalogue that assume a scenario + account structure.
- Non-Planning → read `references/non-planning.md` in full. It is built directly from a real bubble/scatter/quadrant SaaS model and gives you a concrete structural template (tables, relationships, measure patterns, descriptions) for datasets shaped like that one — adapt it to what's actually in front of you rather than forcing the planning catalogue onto it.

Do not mix the two — e.g. do not pull the planning skill's B2B SaaS billing-data measure catalogue (MRR, CAC, churn from an Invoice Line fact table) onto a dataset that has no such fact table just because both happen to be "SaaS." Dataset shape decides the path, not industry label.

If, partway through, the dataset turns out to have elements of both (e.g. a small scenario dimension alongside a chart-showcase fact table), say so explicitly and ask the user how to proceed rather than silently picking one reference file's conventions.

The mode is about *where the schema comes from*, not the dataset shape, and applies the same way regardless of which reference file you loaded above.

```

   AND its CSV output folder and documentation files are accessible
→ MODE 1 — Integrated

ELSE IF the user has uploaded a .pbix file
   OR the user has uploaded a description markdown file
   OR the user explicitly says "model my existing file / uploaded PBIX / description"
→ MODE 2 — Standalone

ELSE

```

**Detection signals for Mode 1:**
- A dataset-creator skill was invoked earlier in this conversation
- CSV files exist in an output folder
- Documentation files (`*_dataset_guide_public.md` and `*_dataset_guide_internal.md`) exist

**Detection signals for Mode 2:**
- A `.pbix` file appears in the uploaded files list

**If neither signal is clear**, ask exactly this — no other preamble:

```
To get started, which of these fits your situation?
```

## CRITICAL BIM Generation Rules — Apply Regardless of Dataset Shape

These rules are the most common failure points for ANY model this skill produces, planning or non-planning. Every BIM — whether by running `build_model.py` or by Claude generating JSON directly — MUST satisfy all of these. Skipping any one causes deployment to fail or produces a broken model.

### Rule 1 — Data Types Must Be Inferred from Column Name Patterns

NEVER read CSV files with `nrows=0` and rely on pandas dtype inference — pandas defaults every column to `object` (string) when there are no data rows to sample.

ALWAYS use an `infer_data_type()` function that pattern-matches the column name:

| Pattern | Type |
|---------|------|
| Ends in `Key` (any case) | `int64` |
| Contains `Date` and not `Update` | `dateTime` |
| Starts with `Is` (any case) | `boolean` |
| Contains `Amount`, `Cost`, `Price`, `Revenue`, `Rate`, `Pct`, `Score`, `Margin`, `Discount`, `Quantity` | `double` |
| Everything else | `string` |

Do not assume `string` as a safe default — it is the most common bug.

### Rule 2 — Hierarchies Are Created Where a Real Drill-Path Exists

Any dimension table with level columns (`Level 1 Name`, `Level 2 Name`, etc.) or a clear drill-path (Year → Quarter → Month, Region → Country) gets a `"hierarchies"` array. Hierarchy names use the `[H] ` prefix; `levels` use display column names (post-rename); ordinals start at 0 for the broadest level.

Not every dataset has hierarchy-worthy dimensions — a flat lookup table with no level structure (e.g. a vendor list with no parent/child grouping) simply gets no hierarchy. Don't force one.

### Rule 3 — Relationship Columns Use Column NAME (Display Name), Not sourceColumn

In the BIM `relationships` array, `fromColumn` and `toColumn` MUST match the column's `name` property (display name after rename) — **NOT** `sourceColumn`. The Tabular Object Model resolves relationship columns by `name`; a mismatch resolves to `(none) ∞←1 (none)` in Tabular Editor.

```json
// CORRECT
{ "fromTable": "[Fact] Financial", "fromColumn": "Account Key", "toTable": "[Dim] Account", "toColumn": "Account Key" }
// WRONG — sourceColumn value used instead of display name
{ "fromTable": "[Fact] Financial", "fromColumn": "AccountKey", "toTable": "[Dim] Account", "toColumn": "AccountKey" }
```

**If the dataset has a `[Dim] Date` table built with the enriched M expression** (see Rule 7), its primary column is named `"Date"`, not `"Date Key"` — so the relationship is asymmetric: `fromColumn: "Date Key"` on the fact side, `toColumn: "Date"` on the dimension side. Datasets with no separate Date table (time lives directly on the fact table) have no such relationship at all — see `references/non-planning.md` for that case.

### Rule 4 — Measure Table Placeholder Column Requires Three Specific Properties

The hidden `_` column in the Measure table is a calculated table column. It MUST have all three of these properties, or Tabular Editor refuses to deploy:

```json
{
  "type":           "calculatedTableColumn",
  "name":           "_",
  "dataType":       "string",
  "sourceColumn":   "[_]",
  "isNameInferred": true,
  "isHidden":       true
}
```

### Rule 5 — Measure Names Must Be Unique Across All Folders

Tabular Editor refuses to deploy when two measures share a name, even across different display folders (`Item 'XXX' already exists in the collection`). Run a duplicate check before writing the BIM:

```python
from collections import Counter
dupes = {n: c for n, c in Counter(m["name"] for m in measures).items() if c > 1}
if dupes:
    raise ValueError(f"Duplicate measure names: {dupes}")
```

### Rule 6 — sortByColumn Must Be Applied to All Label Columns That Need It

Power BI sorts string columns alphabetically by default. Any label column with a corresponding numeric/ordinal column in the same table (Month Short Name → Month Number, Quadrant Label → Quadrant Sort Order, Plan Tier → Plan Tier Sort Order) needs `sortByColumn` pointed at that numeric column. This applies in both dataset shapes — see the relevant reference file's Sort-by-Column Rules for the exact column pairs in that shape.

### Rule 7 — A Separate Date Table, If One Exists, Must Use the Enriched M Expression

**Only applies if the dataset has a standalone `[Dim] Date` / calendar table.** Not every dataset does — see Step 0. When it exists, it must NOT use the simple CSV-load partition every other table uses. It uses an enriched M expression that reads only the raw `Date` column and regenerates ~33 columns from it (offsets, period boundaries, HalfYear, DayType, Prev Day, etc.). The full expression is in `references/planning.md` under "Date Table — Enriched M Expression"; reuse it verbatim for any dataset that has a calendar table, planning or not.

If the dataset instead keeps its time field directly on the fact table (e.g. a `Month Key` column with a paired sort-order column, no separate calendar table), skip this rule entirely — there is nothing to enrich. See `references/non-planning.md` for that pattern.

### Rule 8 — Measures Must Be in a Separate Measure Table

Measures never live inside fact or dimension tables. They live in their own `"Measure"` table with a `DATATABLE` calculated partition and the hidden `_` placeholder column (Rule 4). `write_bim()` should assert a table named `"Measure"` exists and contains at least one measure.

### Rule 9 — DatasetFolder Parameter Must Exist in model.expressions

Every BIM needs a `DatasetFolder` M parameter in `model.expressions`, or no table partition can resolve its CSV path. `write_bim()` should assert this expression exists with the `IsParameterQuery=true` meta annotation. Exact JSON syntax is in `references/planning.md` under "DatasetFolder Expression" — it's identical regardless of dataset shape.

## Shared Output Files (Both Dataset Shapes, Both Modes)

| Output File | Audience | Purpose |
|------------|---------|---------|
| `model.bim` | Tabular Editor / Power BI Desktop | Semantic model — tables, columns, relationships, measures, descriptions |
| `build_model.py` | Developer | Python script that generates all outputs; rerunnable |
| `[name]_model_guide_public.md` | Report developers, demo recipients | Plain-English guide — what pages to build, which measures to use |
| `[name]_model_guide_internal.md` | Logeshkumar Sivakumar (private) | Full technical reference — BIM structure, DAX, relationship map, description sources |
| `measures_reference.md` | Developers | Quick-reference table of every measure: folder, format string, description, full DAX |

**All five files are always generated in one run — never separately, never on request.**

The detailed templates for these documentation files (file headers, section-by-section content, the example "About this model" paragraph, the Use Cases at a Glance table, deployment walkthrough, and the full troubleshooting table) are dataset-shape-agnostic and live in `references/planning.md` under "Model Documentation Files" and "Using Tabular Editor and Power BI Desktop" — read and reuse that section regardless of which shape you're building, substituting whatever folders/measures/use-cases actually apply.

**Deployment steps (same for both modes and both shapes):**
1. Run `build_model.py` → all five files produced
2. Open Tabular Editor 2 or 3 → File → Open from File → select `model.bim`
3. Update the `DatasetFolder` expression to the real CSV path
4. Model → Deploy → select your Power BI Desktop instance (must already be open with a blank report)
5. Power BI Desktop → Home → Refresh to load data
6. Open `[name]_model_guide_public.md` to see what report pages to build next

## Generation Rules Summary (Both Shapes)

| Rule | Requirement |
|------|-------------|
| Table renaming | Every raw CSV name maps to a display name before the BIM is written |
| Column visibility | All FK, PK, and technical columns are hidden |
| Data type inference | Pattern-match column names (Rule 1) — never trust a 0-row pandas read |
| Relationship creation | Only create a relationship if both tables are present |
| Relationship column names | `fromColumn`/`toColumn` use display `name`, never `sourceColumn` (Rule 3) |
| Measure table column | The `_` placeholder needs all three properties from Rule 4 |
| No duplicate measure names | Check before writing the BIM (Rule 5) |
| Measure folders | Numeric prefix; only generate folders relevant to what the dataset actually supports |
| No reconciliation measures | Never compare table totals against each other |
| No debug measures | No `COUNTROWS()` sanity checks, no `_Test`/`_Check`/`_Debug` measures |
| Format strings | Every measure has one — never blank |
| `DIVIDE()` | Always, never the raw `/` operator |
| Variables | Any measure with 2+ computation steps uses `VAR` / `RETURN` |
| Attribution | Both `model.bim` description and `build_model.py`'s top comment carry the attribution block |
